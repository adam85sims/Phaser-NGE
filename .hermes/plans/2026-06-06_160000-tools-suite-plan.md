# Tools Suite — Full Implementation Plan

> **For Hermes:** This is a planning-only document. Use subagent-driven-development to execute.

**Goal:** Design and build a complete tools dashboard and tool suite that gives a content creator everything they need to build a narrative game — project management, scene editing, asset management, layout design, character management, variable tracking, and playtesting — all from one place.

**Architecture:** A hub-and-spoke layout. `tools/index.html` is the control centre dashboard with cards linking to dedicated single-page tools. Each tool is a standalone HTML/CSS/JS app (same pattern as dialogue-editor), all served from the same Vite dev server. Data flows through the `data/` JSON files — tools read via `fetch()`, write via export/download.

**Tech Stack:** Plain HTML + CSS + JS (zero dependencies per tool). Shared CSS theme in `tools/shared/`. No Phaser, no build step, no npm.

---

## 1. User's Mental Model

A content creator opening the tools area for the first time expects:

```
1. A HOME screen showing their project at a glance
2. Quick access to every part of their game
3. To see their stuff — scenes, characters, assets, variables — without digging
4. Everything to feel connected, not like separate apps
```

**The dashboard should answer these questions instantly:**

| Question | Dashboard shows |
|----------|---------------|
| What's my project called? | Project name + description header |
| How big is my game? | Scene count, node count, word count, asset count |
| What did I last work on? | Recently edited scene with timestamp |
| Where do I write dialogue? | Scene Editor card |
| Where do I put my images? | Asset Manager card |
| Where do I add music? | Asset Manager (filtered to audio) |
| How do I set up characters? | Character Manager card |
| How do I set up variables? | Variable Editor card |
| How do I preview my game? | Play Test card (opens game at current scene) |
| How do I change settings? | Project Settings card (gear icon) |

### Cards

Each card is a clickable panel with:
- Icon (emoji or simple SVG)
- Tool name
- Brief stat line (auto-populated from data/)
- Subtle hover effect
- Links to `../tool-name/`

### Stats Bar

Bottom bar auto-calculated by reading `data/` JSON files:
- Scene count: `game.json` scenes array length
- Total nodes: sum of all scene node arrays
- Total words: sum of all dialogue text fields
- Character count: `characters.json` key count
- Variable count: `variables.json` key count
- Asset count: directory scan of `public/assets/` (future)

### Recent Scenes

Pulled from `data/scenes/*.json` — shows:
- Scene name
- Node count
- Word count (sum of all `text` fields)
- "Last edited" (approximated from file metadata or last-export timestamp embedded in the JSON)

## 4. Individual Tool Plans

### 4.1 Asset Manager

**User expectation:** "I need to see all my images, audio, and fonts in one place. I want thumbnails. I want to play audio. I want to know what's a background vs a portrait."

**Features:**
- Browse view: grid of thumbnails for images, file-name list for audio/fonts
- Left sidebar filter: type (background, portrait, icon, BGM, SFX, font), file format
- Search bar: filter by name, type, tag
- Click to preview: larger preview, file info (dimensions, size, duration for audio)
- Drag & drop upload zone (uses File API to stage assets)
- Tag system: assign custom tags per asset
- Usage tracking: which scenes reference this asset? (searches `data/scenes/` for the filename)

**Data source:** Scans `public/assets/` directory structure:
```
public/assets/
├── backgrounds/   → tagged as "background"
├── characters/    → tagged as "portrait"
├── icons/         → tagged as "icon"
├── audio/bgm/     → tagged as "bgm"
├── audio/sfx/     → tagged as "sfx"
└── fonts/         → tagged as "font"
```

For now, since assets may not exist, the tool gracefully shows "No assets yet. Drop files here." and shows placeholder previews for missing files with a "?" icon.

**Export:** Generates an `asset-manifest.json` that the engine can read for preloading.

### 4.2 Character Manager

**User expectation:** "I need to see all my characters, edit their names and colors, assign portrait images, and preview how they'll look in-game."

**Features:**
- Card grid view of all characters from `characters.json`
- Click to edit in right panel:
  - Character ID (internal key)
  - Display name
  - Color picker (nameplate color)
  - Default expression
  - Expression list: add/remove expression keys, assign portrait images
- Portrait preview: shows current expression's portrait (or generated placeholder circle with character color)
- In-game preview: "Show how this character looks in the game" — renders nameplate + portrait on a simulated text box
- "Used in" section: searches scenes for references to this character ID

### 4.3 Variable Editor

**User expectation:** "I need to see all game variables, create new ones, edit defaults, and understand where they're used in my story."

**Features:**
- Table view: Name · Type · Default · Min/Max (for numbers) · Description · Used In
- Inline editing: click cells to edit
- Add variable dialog: name, type (boolean/number/string), default, min, max, description
- Delete with confirmation + usage warning
- "Used In" column: searches `data/scenes/` for references to this variable in conditions, setFlag, setValue
- Filter by type
- Search by name or description

### 4.4 Scene Manager

**User expectation:** "I need to see all my scenes in one place, know how big each one is, and quickly jump to editing."

**Features:**
- List/grid toggle: list view (compact) or grid view (cards with preview)
- Per-scene stats: node count, word count, choice count, character count
- Search bar: filter by scene ID, node content
- Sort: by name, node count, word count
- Click scene → opens in dialogue editor (link to `../dialogue-editor/?scene=xxx`)
- Duplicate scene: creates a copy with "-copy" suffix
- Delete scene: with confirmation
- "Set as start scene" button: updates `game.json` `startScene`

### 4.5 Layout Tool

**User expectation:** "I need to position character portraits on screen, choose backgrounds, and set up where the text box goes — and preview it all together."

**Features:**
- Canvas preview (600×800 mockup of game screen)
- Background selector: dropdown of available background images from assets, shows preview
- Character position anchors: drag characters to preset positions (left, center-left, center, center-right, right)
  - Each position shows the character name + portrait
  - Can toggle visibility per position
- Text box configuration:
  - Position (y-offset from bottom)
  - Width and height
  - Opacity
  - Preset styles: Classic (bottom-center), Full (almost full-width), Minimal (small box, bottom-right)
- Nameplate configuration:
  - Position (above text box, attached, or hidden)
  - Font size
  - Color (inherits from character or custom)
- Choice button configuration:
  - Alignment (left, center)
  - Style (numbered, bullet, none)
- "Export Layout" generates a `layout-config.json` that the engine's DialogueSystem reads
- "Apply to all scenes" or "Apply to current scene only"

### 4.6 Project Settings

**User expectation:** "I need to change my project title, adjust default text speed, and configure the basic look and feel."

**Features:**
- Project name (updates `game.json` title)
- Default text speed slider (show preview: "Hello world" at selected speed)
- Auto-advance default (on/off)
- Default BGM volume, SFX volume
- Background color/theme (hex picker for the game's background)
- Font configuration: select from available fonts (from assets) or use monospace default
- Credits/version info

### 4.7 Play Test

**User expectation:** "I want to launch the game and jump straight to a specific scene to test it."

**Features:**
- Redirects to the game with a query parameter: `/?scene=xxx`
- Engine's `_getStartScene()` checks for URL param, overrides `game.json` `startScene`
- Or opens in a new tab at `/` with the scene selector embedded

## 5. Implementation Tasks

### Phase 1: Foundation

#### Task 1: Create shared theme and utilities

**Objective:** Single CSS theme + JS utility file all tools share, so every tool doesn't duplicate the dark theme styling.

**Files:**
- Create: `tools/shared/theme.css`
- Create: `tools/shared/utils.js`

**theme.css** contains: CSS variables (`--bg`, `--panel`, `--border`, `--text`, `--accent`), base reset, scrollbar styling, common button/input/select styling, card component, table styling.

**utils.js** contains:
```js
window.$ = id => document.getElementById(id);
window.fetchData = async url => { const r = await fetch(url); if (!r.ok) throw new Error(`${url}: ${r.status}`); return r.json(); };
window.setStatus = (el, text, state) => { el.textContent = text; el.className = 'status ' + state; };
window.debounce = (fn, ms) => { let t; return (...args) => { clearTimeout(t); t = setTimeout(() => fn(...args), ms); }; };
```

#### Task 2: Build the dashboard (tools/index.html)

**Objective:** The control centre hub that loads project data and renders cards + stats.

**Files:**
- Create: `tools/index.html`

**Flow:** On load, fetches `game.json`, all scene files, `characters.json`, `variables.json`. Calculates stats. Renders cards and recent scenes list.

**Cards layout:** 2×3 or 3×3 grid depending on screen width. Each card is a `<div>` with icon, name, stat, hover effect, and `<a>` link to the tool's URL.

**Stats:**
```js
async function loadStats() {
  const game = await fetchData('/data/game.json');
  const chars = await fetchData('/data/characters.json');
  const vars = await fetchData('/data/variables.json');
  let totalNodes = 0, totalWords = 0;
  const scenes = [];
  for (const id of (game.scenes || [])) {
    try {
      const scene = await fetchData(`/data/scenes/${id}.json`);
      totalNodes += scene.nodes?.length || 0;
      totalWords += countWords(scene.nodes);
      scenes.push({ id, nodes: scene.nodes?.length || 0, words: countWords(scene.nodes) });
    } catch(e) { /* scene not found */ }
  }
  // Render stats and cards
}
```

**Word count helper:**
```js
function countWords(nodes) {
  return (nodes || []).reduce((sum, n) => sum + (n.text ? n.text.split(/\s+/).filter(Boolean).length : 0), 0);
}
```

## 6. Data Flow Architecture

```
                        ┌──────────────────┐
                        │   data/game.json  │
                        │   characters.json │
                        │   variables.json  │
                        │   scenes/*.json   │
                        └────────┬─────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
     ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
     │   Dashboard   │   │ Individual   │   │   Game       │
     │ (tools/index) │   │   Tools      │   │ (src/engine) │
     │   reads all   │   │  read/edit   │   │  reads via   │
     │   for stats   │   │  per-type    │   │  Data store  │
     └──────────────┘   └──────┬───────┘   └──────────────┘
                               │
                        Export/Download
                               │
                               ▼
                    User places file in data/
                    (or File System Access API)
```

**Key insight:** Tools read via fetch, write via download. The user is the "write path." This keeps everything simple — no backend, no auth, no database.

## 8. Risks & Open Questions

| Risk | Mitigation |
|------|------------|
| **No directory listing** — asset manager can't list files in `public/assets/` from the browser | Use a pre-built manifest file, or add a small Vite middleware endpoint. For MVP, hardcode the asset structure. |
| **Stats page load time** — fetching every scene file to calculate word counts is slow for 50+ scenes | Cache stats in localStorage with a 30-second TTL. Show cached stats immediately, update in background. |
| **Tool fragmentation** — 8 separate HTML pages feel disconnected | Shared theme.css makes them visually consistent. The dashboard is the glue. Add a persistent nav bar to every tool page. |
| **Export/download friction** — user has to manually place files | File System Access API as progressive enhancement. Download as primary. Add a "Save to data/" button that guides the user. |
| **Writer's block** — tools are great, but they don't help with actually writing | Add a "random prompt" or "scene starter" widget to the dashboard. Very low effort, high morale impact. |

## 9. Open Questions for the User

1. **Dashboard first, or a specific tool?** Should we build the dashboard hub first (so every tool has a home), or build individual tools and add the dashboard later? Dashboard-first gives immediate structure.

2. **Asset directory scanning** — do you want to set up a simple directory structure in `public/assets/` first (create some folders, maybe drop in a test image or audio file), so the asset manager has something to work with?

3. **One editor to rule them all?** Instead of separate tools for scenes, characters, variables — should there be a single "project editor" with tabs/panels for everything? The separate-tools approach keeps each page focused, but tabs reduce context switching.

4. **Persistent nav bar** — should every tool page have a top bar with the project name + links back to dashboard and other tools? This makes the suite feel cohesive.

5. **What's your priority?** After the dialogue editor, which tool would you use most? Character manager? Variable editor? Scene manager?
